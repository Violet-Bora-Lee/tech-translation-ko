# tech-translation-ko 설계 문서

작성일: 2026-07-18

## 목적

ko.javascript.info 저장소에서 이보라(Violet Bora Lee, Bora Lee, Violet.Lee 명의) 커밋의 번역·교정을 전수 분석해 영→한 기술문서 번역 기법을 추출하고, Claude Code Skill 겸 범용 가이드로 만들어 오픈소스화한다. 로컬에 en.javascript.info와 ko.javascript.info가 나란히 있어(`/Users/bora/experiments/js-info/`) 원문 대조가 가능하다.

## 분석 입력 소스

1. **신규 번역 diff**: 이보라 커밋이 파일 내용의 대부분을 만든 문서. en 원본과 문단 대조.
2. **교정 리뷰 diff**: 타인 번역을 이보라가 고친 diff. '의미 오류 수정 / 문체 통일 / 용어 변경 / 원문 갱신 대응 / 판정 불가' 다섯 범주로만 제한 코딩한다.
3. **wiki·CONTRIBUTING**: 번역 모범 사례, 용어집 등 명문화된 규칙.
4. **운영 스프레드시트**: 프로젝트 운영 구글 시트의 전 시트를 Chrome 자동화로 순회 수집. 진행 관리 시트(리뷰현황, 본문번역현황 등)는 제외하고 용어집·표기 규칙·논의 기록만 수집한다.

## 분석 파이프라인

### 1단계, 코퍼스 manifest 생성 (스크립트)

추출·분류·정렬을 한 단계로 통합한 스크립트가 파일별로 기록한다.

- 기여자 신뢰도: 확정(이보라 커밋이 본문 대부분 작성) / 개연(부분 기여)
- 변경 유형: 신규 번역 / 교정 리뷰 / 혼합
- 번역 시점과 가까운 en 리비전(가능한 경우). en·ko 시점 불일치 완화 목적
- 제외 사유(제외 파일의 경우)

### 2단계, 코드북 선행 + 병렬 정밀 분석

- 대표 문서 2~3편을 메인 에이전트가 직접 분석해 패턴 분류 체계(코드북)를 만든다.
- 신규 번역 문서를 챕터·분량 기준 청크로 나눠 병렬 서브에이전트가 코드북 기준으로 en↔ko 문단 대조 후 구조화 보고(패턴명·원문·번역·설명·빈도). 일부 청크는 중복 배정해 일관성을 확인한다.
- 관찰 축은 문장 분해·어순 재배치, 어휘 선택(번역·음차·병기), 어미·문체, 코드 주석·캡션·과제 처리, 생략·보강.
- 교정 diff는 별도 에이전트가 다섯 범주 제한 코딩으로 분석한다.

### 3단계, 병합·검증

- 패턴을 증거표(사례·반례·빈도·출처·신뢰도)로 통합한다.
- 저빈도·상충 패턴은 원본 diff로 재검증한다.
- wiki·시트의 '명문 규칙'과 diff에서 발굴한 '암묵지'를 출처 표기로 구분한다.

## 스크립트 원칙

결정적으로 판별 가능한 규칙은 전부 스크립트로 내린다. LLM은 기계 판별 불가 영역(문장 분해, 자연스러움, 의미 보존)에 집중한다.

`check_translation.py` 검사 항목(분석 결과에 따라 확장)과 등급.

- **error(확실)**: em-dash, 문장 끝 쌍점, 헤딩 끝 문장부호, 명사 종결 마침표, 마크다운 구조 불일치(코드블록 수·헤딩 레벨·링크)
- **warning(검토 필요)**: 이중 피동 후보, 번역투 패턴, 용어집 위반 후보, 약어 풀이 누락, 가운뎃점 누락

출력은 라인 번호 + 규칙 ID 리포트. 기본 워크플로우는 '번역 → 스크립트 검사 → 위반 수정 → 재검사' 루프.
