# Principles · 번역 철학과 우선순위

What "good" means in this skill, and how to trade off when rules collide.
규칙이 충돌할 때의 판단 기준이다.

## The goal · 목표

번역문은 '영어를 옮긴 글'이 아니라 '처음부터 한국어로 쓴 좋은 기술 글'처럼 읽혀야 한다.
기준은 세 가지다.

1. **오역이 없다.** 의미 누락도 오역이다.
2. **번역투가 없다.** 단어-단어 대응은 목표가 아니다.
3. **독자가 원문 독자보다 손해 보지 않는다.** 원문 독자가 문맥으로 아는 배경은
   보강해서라도 채워 준다.

## Priority order · 우선순위

충돌 시 위가 이긴다.

1. **의미 보존** — 재구성·보강은 자유지만 의미가 바뀌면 실패다.
2. **용어 일관성** — 용어집이 개인 취향을 이긴다. 공동 작업에선 특히 그렇다.
3. **구조 계약** — 줄 수·마크다운 구조 보존이 요구되는 프로젝트라면 문장 재구성도
   그 제약 안에서 한다.
4. **한국어다움** — 위 세 가지를 지킨 다음에 문장을 한국어답게 다듬는다.
5. **원문의 형식** — 가장 낮은 우선순위다. 원문의 문장 수·어순·구두점은 보존 대상이
   아니라 재료다.

## Practice beats doctrine · 명문 규칙보다 실제 관행

규칙 문서와 실제 코퍼스가 다르면 코퍼스가 답이다. 이 skill을 만들 때도 wiki가
금지한 헤딩 물음표를 실제 번역('# 자바스크립트란?')이 쓰고 있어 허용으로 조정했다.
사용 중 규칙이 어색하면 실제 좋은 번역 사례를 찾아 규칙을 고쳐라. 같은 원리로,
권위 있는 근거로 정한 역어라도 실제 문장에서 어색하면 재론한다(custom → '사용자
지정'을 MS 용어집 근거로 정했다가 실사용 후 '커스텀' 혼용으로 변경한 사례).

## Reader model · 독자 모델

기술 튜토리얼 독자는 '배우는 중'이다. 그래서 이 코퍼스의 번역자는 일관되게
독자 쪽으로 기운다.

- 압축된 원문을 풀어 자기완결적으로 만들고
- 실제 화면·출력과 어긋나는 서술을 사실로 교정하고
- 처음 보는 독자를 가정한 조건('처음 열어보셨다면')을 신설하고
- 강연하듯 말을 건다('-죠', '살펴봅시다')

API 레퍼런스나 명세처럼 독자가 '찾아보는 중'인 문서라면 이 중 정서적 장치는 줄이고
정확·간결로 기운다. 문서 장르에 맞게 다이얼을 조절한다.

## Workflow discipline · 작업 규율

- 번역 전: 원문을 끝까지 읽고, 용어집을 스캔한다(`glossary_lookup.py scan`).
- 번역 중: 새 용어는 등재 후 사용한다. 모호한 원문은 임의 해석하지 않는다.
- 번역 후: 린터 실행 → error 수정 → warning 판단 → 맞춤법 검사기.
- 결정을 바꿀 땐 근거와 함께 이력을 남긴다.
