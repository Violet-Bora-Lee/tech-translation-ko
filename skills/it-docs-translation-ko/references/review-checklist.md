# Review Checklist · 번역 검수 체크리스트

Built from limited-category coding of 135 real correction commits (1,155 hunks) in a
large translation corpus — what an experienced reviewer actually fixes, ordered by
observed frequency. 숙련 리뷰어가 실제로 가장 자주 고친 것부터 점검한다.

린터(`check_translation.py`)가 잡는 항목은 (lint) 표시. 나머지는 사람·LLM 판단 영역이다.

## 0th pass — 원문 최신본 확인 (번역 시작 전)

- [ ] 번역 대상 스냅샷이 원문 최신본과 같은가. 원문 저장소가 계속 갱신되는
  프로젝트라면 로컬 스냅샷이 뒤처져 있기 쉽다. `diff <로컬 원문> <최신 원문>`
  한 번이면 되고, 다르면 최신본 기준으로 번역한다. 실제로 스냅샷이 구버전 API
  (`FileReader`) 기준이었는데 최신 원문은 신규 API(`blob.arrayBuffer()`)로 바뀌어
  있던 사례가 있다.

## 1st pass — 기계 검사 (린터 실행)

- [ ] em-dash, 문장 끝 쌍점, 헤딩 문장부호, 괄호·마침표 위치 (lint: error)
- [ ] 원문 대비 줄 수·코드블록·헤딩·링크 구조 일치 (lint: error, `--source` 필요.
  구조 계약을 채택한 프로젝트만)
- [ ] 짝 단어 가운뎃점: 대소문자·대/소문자 → 대·소문자 (lint: error)
- [ ] 강조용 큰따옴표 → 작은따옴표 (lint: warning. 교정 코퍼스 최다 반복 유형 2위)

## 2nd pass — 표기·용어 일관성

- [ ] **맞춤법 검사기를 돌렸는가.** 교정 코퍼스 최다 반복 유형 1위가 오탈자 일괄
  교정이다(대게→대개 32회, 쳅터→챕터, 클로져→클로저, 제너레티어→제너레이터).
- [ ] 용어집 위반이 없는가 (lint: warning + `glossary_lookup.py`). 한 문서 안에서
  같은 개념에 두 역어를 쓰지 않았는가(연결/커넥션, 수학 연산/산술 연산).
- [ ] 자료형류 합성어 표기 통일: 숫자형·문자형·객체형(분리 표기 '숫자 형' 금지)
- [ ] 조사 오류: 앞엔/앞에, 에서/에 같은 미세 조사 확인. 코드 식별자 뒤 조사는
  읽는 소리 기준으로 붙인다(`highlightElem`을 / `highlightElement`를). 식별자를
  고치면 조사도 다시 확인한다.
- [ ] 의존명사 띄어쓰기: 횟수를 뜻하면 '한 번'으로 띄어 쓴다('한번 시도해 보다'처럼
  '어디 한번'의 뜻일 때만 붙임).
- [ ] 영어 병기 과잉: 핵심 개념 첫 등장만 병기하고, 보조 개념어의 병기는 삭제한다
  (숫자형(Number) → 숫자형). 헤딩의 병기·따옴표도 제거한다(# 객체(object) → # 객체,
  ## 'get' 트랩 → ## get 트랩).

## 3rd pass — 문장 품질 (교정 코퍼스의 '문체 통일' 300건에서 추출)

- [ ] 번역투 명사구가 남아 있지 않은가. "3 가지 솔루션 변형 만들기" →
  "아래 방법을 사용해 세 가지 답안을 만들어보세요."
- [ ] 긴 복문을 분해했는가. 반대로 장황한 두 문장은 한 문장으로 압축했는가.
- [ ] 주어가 문장 뒤에 밀려 있지 않은가(주어를 앞으로).
- [ ] 불필요한 '만일/만약', 중복 명사('파일의 줄 수'→'줄 수'), 잉여 쉼표 제거
- [ ] 지시 표현을 구체 대상으로: "위에서 보았듯이" → "예시에서 보았듯이"
- [ ] 어색한 관형사형 인용을 서술형으로: "'다시 열' 방법이 없습니다" →
  "다시 여는 방법이 없습니다"
- [ ] 피동 남용: "더 빠른 웹 페이지를 만들 수 있습니다" → "실 속도가 빨라집니다"
- [ ] **행위 주체·대상이 문면에 드러나는가.** 검수 세션에서 가장 자주 나온 독자 질문이
  "받는 주체가 뭐야?", "응답하는 주체는 어디야?"였다. 피동문("전달됩니다",
  "호출됩니다")은 '누가 누구에게'가 사라지기 쉬우니 브라우저가·서버가·콜백이 같은
  주체를 복원할 수 있는지 검토한다(sentence-patterns 10번).
- [ ] 앞 문장과의 논리 연결이 끊긴 곳은 없는가. 독자가 '왜 갑자기 이 이야기가
  나오지?'라고 물을 지점은 어순 재배열이나 접속어로 연결 고리를 드러낸다.
- [ ] 장르에 맞는 어조인가. 튜토리얼이면 건조한 직역 서술에 독자 톤을 입힌다
  ("아래는 ~에 대한 설명입니다." → "마지막 세 줄은 약간의 설명이 필요해 보이네요.").
  레퍼런스·명세면 반대로 정서적 장치를 걷어낸다.

## 4th pass — 의미 검증 (교정 코퍼스 '의미 오류 수정' 147건)

- [ ] 코드와 설명이 일치하는가. 실제 출력값을 주석에 명시했는가
  (`// true(if 조건문이 끝났어도 변수에 여전히 접근할 수 있음)`).
- [ ] 예제를 실제로 실행해 확인했는가. 렌더링 결과·실동작까지 검증하면 원문 설명과
  실동작이 어긋나는 부분(파라미터 기본값 오기 등)을 잡을 수 있다.
- [ ] 번역한 코드 문자열(UI 메시지·에러 인자)을 grep해 본문의 백틱 참조와
  일치하는지 확인했는가. 린터는 반쪽 현지화를 잡지 못한다.
- [ ] 좌표·방향·순서처럼 뒤바뀌기 쉬운 쌍(clientX/pageX 류)을 원문과 재대조했는가.
  실제 코퍼스에서 의미 역전 오역이 발견됐다.
- [ ] 원문 갱신 대응: 번역 후 원문이 바뀌지 않았는지 diff를 확인한다.
- [ ] **원문 자체의 사실 오류를 그대로 옮기지 않았는가.** 본문 API 이름이 실제
  라이브러리·문서 내 데모 코드와 일치하는지, 예시 코드에 문법 오류(쉼표 누락 등)가
  없는지 검증한다. 원문 오류를 발견하면 번역본에서 바로잡고 원문과 달라진 부분과
  이유를 기록한다. 원문 수정은 원문 저장소 이슈·PR로 분리해 보고한다. 실제 사례:
  원문의 `Prism.highlightElem`은 존재하지 않는 API였다(실제는 `highlightElement`).
- [ ] 누락 문장이 없는가. 의역은 허용되지만 누락은 불허.

## House style · 하우스 스타일 결정 이력

프로젝트마다 굳혀야 할 결정 목록이다. 정답은 없고, 정한 뒤 일관 적용과 기록이 정답이다.

- 예제 에러·인사 문자열의 표준 치환("Whoops!" → "에러 발생!" 류)
- 예제 인명·문화 레퍼런스의 현지화 여부와 수위
- 원어 유지 용어 목록(getter/setter를 음차할지 원어로 둘지 등)
- 검색 유입을 고려한 어휘 선택('기초'보다 '기본' 같은 결정은 근거를 커밋 메시지에 남긴다)

## Reviewing others · 타인 번역을 리뷰할 때

- 수정은 범주를 의식하고 한다: 의미 오류 > 용어 통일 > 문체 통일 순으로 우선한다.
- 용어를 일괄 치환할 땐 커밋 메시지에 '[용어 변경] X → Y'와 근거를 남긴다.
- 문체 취향 교정과 오류 교정을 한 커밋에 섞지 않으면 리뷰가 쉬워진다.
